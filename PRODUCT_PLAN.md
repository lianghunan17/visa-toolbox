# 签证工具箱完整落地版

## 1. 最终产品定义

- 商品名：签证工具箱
- 售卖方式：一次付费
- 授权期限：永久使用
- 授权形式：唯一激活码
- 默认设备限制：1 台
- 备用规则：支持后台人工换绑
- 当前阶段定位：最小可卖版

这不是共享密码产品，而是标准数字商品授权产品。

---

## 2. 参考行业做法后的结论

类似数字工具产品，主流做法基本是：

1. 支付平台收款
2. 自动发 license key 或激活码
3. 用户首次输入激活码
4. 服务端完成激活和设备绑定
5. 后续继续做轻量验权
6. 管理后台支持停用、重发、重置绑定

适合你的不是“统一密码”，而是“一人一码”。

---

## 3. 你这个产品的推荐售卖规则

### 售卖规则
- 用户一次购买
- 获得一个永久授权码
- 授权码默认只能绑定 1 台设备
- 如更换设备，由后台手动重置设备绑定

### 风控规则
- 不允许多人共用一个码
- 不暴露管理员密码
- 不把验权逻辑写死在前端
- 服务端保留停用单个授权的能力

### 用户承诺文案建议
- 一次购买，永久使用
- 默认支持 1 台设备激活
- 更换设备可联系客服协助换绑
- 工具结果仅供参考，不构成法律意见

---

## 4. 推荐技术架构

### 起步版推荐组合
- 收款：Lemon Squeezy 优先，其次 Stripe
- 前端：当前 `pr-app/` 静态站
- 授权接口：Vercel Serverless 或 Cloudflare Workers
- 数据库：Supabase
- 后台：先做轻量管理页，后续再增强

### 为什么这样选
- 前期开发量小
- 前期成本低
- 可直接利用免费额度
- 容易迁移，不会一开始就被复杂基础设施拖死

---

## 5. 免费部署这件事，现实做法

“永久免费部署”不适合当成绝对承诺。

更合理的目标是：

- 用户永久使用
- 你的服务端前期尽量低成本
- 早期跑在免费额度
- 后期有收入后再升级
- 数据结构可迁移，不锁死平台

### 推荐现实方案
- 静态前端：Cloudflare Pages 免费层
- 授权接口：Vercel 免费层或 Cloudflare Workers 免费层
- 数据库：Supabase 免费层
- 支付通知：由支付平台 webhook 驱动

## 6. 用户完整流程

### 购买流程
1. 用户进入购买页
2. 完成支付
3. 支付成功后生成授权记录
4. 用户收到激活码和使用说明

### 首次使用流程
1. 用户打开签证工具箱
2. 页面检测是否已激活
3. 未激活则跳转到激活页
4. 用户输入邮箱 + 激活码
5. 服务端验证订单和授权状态
6. 验证通过后绑定设备
7. 返回授权 token
8. 前端保存本地激活状态

### 后续使用流程
1. 用户访问工具页
2. 前端检查本地授权信息
3. 关键功能前调用验权接口
4. 服务端返回有效 / 失效 / 停用 / 需重新激活
5. 前端按状态处理

## 9. 接口设计

## 1. 激活接口
`POST /api/license/activate`

请求：
- email
- licenseKey
- deviceId
- deviceName

校验：
- license 是否存在
- email 是否匹配购买邮箱
- status 是否可用
- 是否超过设备数

返回：
- success
- authToken
- licenseStatus
- productName
- activationUsage

---

## 2. 验权接口
`POST /api/license/validate`

请求：
- authToken
- deviceId
- toolName

返回：
- valid
- status
- message

用途：
- 页面进入时
- 核心功能前
- 定时静默校验

---

## 3. 退出 / 清除本地授权
`POST /api/license/deactivate-local`

说明：
- 这个接口主要用于前端本地清理和用户主动解绑提示
- 真正解绑可由管理员处理

---

## 4. 管理后台接口

### 新建授权
`POST /api/admin/licenses`

### 查询授权
`GET /api/admin/licenses`

### 查看详情
`GET /api/admin/licenses/:id`

### 停用授权
`POST /api/admin/licenses/:id/disable`

### 重置设备绑定
`POST /api/admin/licenses/:id/reset-device`

### 重发激活信息
`POST /api/admin/licenses/:id/resend`

## 10. 设备绑定策略

### 起步建议
- 默认 1 台设备
- 首次激活后记录 device_id
- 同一设备可重复验证
- 新设备激活时提示“已超过设备限制”
- 后台支持人工重置

### device_id 怎么做
前端可基于以下组合生成稳定指纹：
- userAgent
- language
- timezone
- 屏幕尺寸
- 本地随机 seed

说明：
- 浏览器环境里设备识别无法绝对可靠
- 起步阶段够用就行，不必追求银行级风控

---

## 11. 前端接入原则

当前 `pr-app/` 是纯静态前端，意味着：
- 任何前端代码都能被用户看到
- 所以不能把核心授权逻辑放前端

正确做法：
- 前端只负责收集输入
- 前端只保存短 token 或激活状态
- 真正校验由后端完成

## 14. 对你当前项目的直接改造建议

当前 `pr-app/` 已经是一个可公开部署的静态工具箱。

所以推荐分两步走：

### 第一步
先保持前端结构不大改，只补：
- 激活页
- 授权守卫脚本
- 首页授权状态提示

### 第二步
再接上：
- 后端授权接口
- 数据库
- 支付 webhook

这样不会一下把现有项目推倒重来。

- 一人一码

- 承诺“用户购买后长期可用”
