# 公开实施清单

这份清单面向当前 `pr-app/` + `license-api/` 项目，目标是把“本地可用”推进到“可稳妥公开”。

## 一，发布策略建议

当前建议：
- 先公开为 **公测版**，不要直接走收费正式版
- 先灰度给少量真实用户，再逐步放大

原因：
- 刚完成运行链路修复
- 诊断系统已经具备，但仍值得经历一轮真实用户验证
- 反馈闭环越早跑起来，后面改动越稳

---

## 二，发布前必须完成

### 1. 后端必须通过自检

在 `pr-app/license-api` 执行：

```bash
npm run preflight
```

要求：
- 必须通过
- 不允许残留测试地址、默认 token、TLS 绕过配置

### 2. 后端运行检查必须变绿

检查：

```bash
curl http://127.0.0.1:8787/health
curl http://127.0.0.1:8787/ready
```

要求：
- `health.ok = true`
- `ready.ok = true`

### 3. 后台页面必须可用

打开：

- `admin.html`

要求：
- 诊断面板能正常显示
- API 连通性正常
- 配置状态正常
- 数据库状态正常
- 管理员鉴权正常
- TLS 状态正常

### 4. 前端不能保留本地地址

公开前检查：
- 前端页面不能继续指向 `127.0.0.1`
- 公开环境必须显式注入正式 API 地址

当前本地联调用法：

```html
<script>
  window.__LICENSE_API_BASE__ = 'http://127.0.0.1:8787';
</script>
```

正式公开时必须改成正式 API 域名。

### 5. 保留免责声明

建议保留：

> 本工具基于日本出入国在留管理庁公开资料与整理规则制作，仅供参考，不构成法律意见或官方承诺。请以最新官方信息及实际审查结果为准。

---

## 三，正式部署顺序

### 路线 A，推荐

- 静态前端：Cloudflare Pages
- API：独立 Node 运行环境
- 数据库：Supabase

### 第一步，部署 API

最少要准备：
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `ADMIN_BEARER_TOKEN`
- `LICENSE_TOKEN_SECRET`
- `CORS_ORIGIN`

要求：
- `SUPABASE_SERVICE_ROLE_KEY` 只能放后端
- `LICENSE_TOKEN_SECRET` 必须是长随机字符串
- `ADMIN_BEARER_TOKEN` 不能泄露到前端
- `CORS_ORIGIN` 必须改成正式前端域名

部署后立刻检查：

- `https://你的-api-domain/health`
- `https://你的-api-domain/ready`

要求：
- 两个接口都返回正常

### 第二步，部署静态前端

直接部署 `pr-app/` 到 Cloudflare Pages。

重点：
- `admin.html` 仅用于你自己管理，不建议公开导航暴露
- 公开页只放用户真正需要的入口
- 正式环境必须注入：

```html
<script>
  window.__LICENSE_API_BASE__ = 'https://你的-api-domain';
</script>
```

### 第三步，前后端联调

最少验证：
- 首页打开正常
- `/visa` 正常
- `/pr` 正常
- 管理后台正常
- 创建授权正常
- 激活授权正常
- 验证授权正常
- 反馈提交正常

---

## 四，公开当天的实施顺序

### 阶段 1，灰度

先给 3 到 10 个真实用户：
- 看页面是否正常打开
- 看是否出现意外报错
- 看反馈入口是否被真正使用

### 阶段 2，小范围扩散

如果灰度没问题：
- 发到熟人群
- 发到小范围社群
- 收第一波真实反馈

### 阶段 3，正式公开

如果前两阶段稳定：
- 再发公开链接
- 再扩大访问量

---

## 五，回滚预案

公开前就要准备好：

### 最小回滚策略

1. 前端切回测试公告
2. 暂停收费入口或激活入口
3. 保留后台和诊断接口
4. 如果 API 出问题，优先让页面展示“维护中”，不要静默失败

### 你至少要能做到

- 快速切换站点模式
- 快速替换前端公告文案
- 快速停用公开入口

---

## 六，公开后第一天重点盯什么

1. `/ready` 是否持续为绿
2. 是否出现数据库连接失败
3. 是否出现管理员鉴权异常
4. 是否出现用户端激活失败
5. 是否有明显反馈说“看不懂 / 不敢用 / 结果不可信”

如果有问题，优先级建议：
1. 可用性问题
2. 错误提示问题
3. 用户理解成本问题
4. 功能增强

---

## 七，当前项目的直接行动建议

按现在这套代码，建议你这样做：

1. 先把 API 部署到正式域名
2. 确认正式 `/health` 和 `/ready` 变绿
3. 再把前端部署到 Cloudflare Pages
4. 把前端里的 API 注入改成正式域名
5. 先做公测公开，不直接收费
6. 先灰度一小批用户

---

## 八，发布完成定义

满足以下条件，才算真的可以公开：

- `npm run preflight` 通过
- 正式 API 的 `/health` 为绿
- 正式 API 的 `/ready` 为绿
- 前端已切到正式 API 域名
- 后台诊断面板显示正常
- 授权创建 / 激活 / 验权至少走通一遍
- 免责声明保留
- 有回滚办法

如果缺任何一项，都不建议直接大范围公开。
